# Migration 0009 — Vendor CLAUDE.md workflow block

Brings projects from AgenticApps workflow v1.6.0 to v1.8.0 by replacing the
inlined ~150-line workflow boilerplate in CLAUDE.md with a vendored
`.claude/claude-md/workflow.md` file plus a short reference block.

The inlined block was originally appended by `migrations/0000-baseline.md`
Step 4 (which `cat`-ed `templates/claude-md-sections.md` directly into
CLAUDE.md). Audit on 2026-05-13 found two consumer repos blowing the
200-line CLAUDE.md context budget because of this:

- `factiv/cparx/CLAUDE.md` — 646 lines (3.23× over)
- `factiv/fx-signal-agent/CLAUDE.md` — 372 lines (1.86× over)

`fx-signal-agent`'s CLAUDE.md retains the source template's H1 line
verbatim (`# CLAUDE.md Sections — paste into your project's CLAUDE.md`),
proving the content was pasted unchanged. ADR 0021 captures the rationale
for the inline → vendor pivot.

This migration is paired with a fix to `migrations/0000-baseline.md`
Step 4 (committed in the same release) so fresh installs vendor from
the start; 0009 handles the upgrade path for existing 1.6.0 projects.

## Pre-flight

```bash
# Required: project at exactly v1.6.0 (or update skill chains earlier
# migrations first — 0008 takes projects from 1.5.0 to 1.6.0)
INSTALLED=$(grep -E '^version:' .claude/skills/agentic-apps-workflow/SKILL.md | sed 's/version: //')
test "$INSTALLED" = "1.6.0" || { echo "ERROR: installed version is $INSTALLED, this migration requires 1.6.0"; exit 1; }

# Required: project files exist (this migration patches them, doesn't create them from scratch)
test -f CLAUDE.md || { echo "ERROR: CLAUDE.md not found — was 0000-baseline applied?"; exit 1; }
test -f .claude/skills/agentic-apps-workflow/SKILL.md || { echo "ERROR: workflow skill missing — re-run /setup-agenticapps-workflow"; exit 1; }

# Required: vendored template source available in the workflow scaffolder
test -f ~/.claude/skills/agenticapps-workflow/templates/.claude/claude-md/workflow.md \
  || { echo "ERROR: workflow scaffolder is older than 1.8.0 — git pull on ~/.claude/skills/agenticapps-workflow"; exit 1; }
```

## Steps

### Step 1: Create `.claude/claude-md/workflow.md` from canonical template

**Idempotency check:** `test -f .claude/claude-md/workflow.md`
**Pre-condition:** the source template exists at `~/.claude/skills/agenticapps-workflow/templates/.claude/claude-md/workflow.md`
**Apply:**
```bash
mkdir -p .claude/claude-md
cp ~/.claude/skills/agenticapps-workflow/templates/.claude/claude-md/workflow.md \
   .claude/claude-md/workflow.md
```
**Rollback:** `rm -f .claude/claude-md/workflow.md && rmdir .claude/claude-md 2>/dev/null || true`

### Step 2: Verify vendored content is current (divergence detection)

**Idempotency check:** `grep -qE "^#{2,4} Superpowers Integration Hooks \(MANDATORY" .claude/claude-md/workflow.md`
**Pre-condition:** Step 1 succeeded — file exists
**Apply:** No-op when the idempotency check passes (the marker is the
canonical content sentinel — its presence proves the vendored file
contains the current workflow block). When the check fails (file exists
but content is older or has been customised by the user), the migration
runtime emits a divergence prompt:

> `.claude/claude-md/workflow.md` exists but does not match the v1.8.0
> canonical template. Choose:
> A) Replace with v1.8.0 canonical (overwrites local edits)
> B) Keep local copy (skip this step; you'll need to re-sync manually if
>    the workflow scaffolder ships a substantively new revision)
> C) Show diff first

The user-facing diff is generated by `diff -u
~/.claude/skills/agenticapps-workflow/templates/.claude/claude-md/workflow.md
.claude/claude-md/workflow.md`.

The heading-level-agnostic regex (`^#{2,4}`) catches both the new
canonical (`## Superpowers...`, H2) and the legacy inlined shape
(`### Superpowers...`, H3, as emitted by the deprecated
`templates/claude-md-sections.md` source) so this Step 2 check correctly
treats either as "vendored content current".

**Rollback:** the migration runtime commits **once per migration**
(see `update/SKILL.md` Step 5 #6), not once per step. While 0009 is
running, no commit has happened yet — Step 2's "Apply replaced the file"
work is sitting in the working tree alongside Step 1's `cp`. To roll
back Step 2 specifically, the runtime restores the working-tree state
captured by an in-memory snapshot taken at Step 2 entry. If the runtime
opts to abort the entire migration (per `update/SKILL.md` Step 5 #8),
the documented effect is to roll back **all** of Steps 1, 2, 3, 4 to
the pre-0009 disk state (the prior migration's commit), not just Step 2.
A per-step rollback that preserves earlier in-flight steps requires the
runtime's snapshot facility; without it, abort-and-restart is the only
guaranteed-safe path.

### Step 3: Add reference block to CLAUDE.md

**Idempotency check:** `grep -q "claude-md/workflow.md" CLAUDE.md`
**Pre-condition:** `CLAUDE.md` exists
**Apply:** append a short reference block to CLAUDE.md (above any
pre-existing inlined block; Step 4 handles the inlined block's removal):

```bash
cat >> CLAUDE.md <<'EOF'

## Workflow

This project uses the AgenticApps Superpowers + GSD + gstack workflow.
Full hooks, rituals, and red-flag tables: [`.claude/claude-md/workflow.md`](.claude/claude-md/workflow.md).
Vendored — re-sync via `/update-agenticapps-workflow`.
EOF
```

**Rollback:** delete the appended `## Workflow` section from CLAUDE.md.
Anchor: the block starts with `## Workflow` on a line by itself and ends
with the line `Vendored — re-sync via `/update-agenticapps-workflow`.`.

### Step 4: Detect and extract pre-existing inlined block

**Idempotency check:** `! grep -qE "^#{2,4} Superpowers Integration Hooks \(MANDATORY" CLAUDE.md`
**Pre-condition:** Step 3 succeeded (reference block added — extraction is
safe because the canonical content is now reachable via the vendored file)
**Apply:** the migration runtime evaluates the inlined-block detection
signature and decides what to do:

```bash
# Detection: any of these markers in CLAUDE.md indicates an inlined block.
# The Superpowers heading regex is heading-level-agnostic (#{2,4}) because:
# - The deprecated `templates/claude-md-sections.md` source emits H3
#   (### Superpowers Integration Hooks ...) — this is what cparx and
#   fx-signal-agent have on disk today.
# - The new vendored canonical `templates/.claude/claude-md/workflow.md`
#   emits H2 (## Superpowers Integration Hooks ...).
# - A future template revision may renumber further; the broad #{2,4}
#   range tolerates that without code changes.
INLINED=0
grep -qE "^#{2,4} Superpowers Integration Hooks \(MANDATORY" CLAUDE.md && INLINED=1

# Smoking-gun marker — proves the block was pasted verbatim from the
# deprecated template (only fx-signal-agent's CLAUDE.md retains this H1).
# When found, it ALSO promotes INLINED to 1, so even projects whose
# Superpowers heading was renamed during customisation still get detected.
PASTED_VERBATIM=0
grep -qE "^# CLAUDE.md Sections [—-] paste into your project's CLAUDE.md" CLAUDE.md \
  && PASTED_VERBATIM=1 \
  && INLINED=1
```

The em-dash `[—-]` character class accepts both U+2014 (em-dash, the
template's original byte) and U+002D (hyphen-minus, the most common
editor-save normalisation), so a CLAUDE.md whose punctuation was
normalised by an editor still trips the smoking-gun detector.

If `INLINED=0`, this step is a no-op (the project never inlined the
content — it's already clean).

If `INLINED=1`, the runtime determines the extraction range:

- **Start line:** first match (in CLAUDE.md, in this priority order) of:
  1. `^# CLAUDE.md Sections [—-] paste into your project's CLAUDE.md` (deprecated H1; if present, this is the absolute start — the line itself and everything after it through the end anchor is part of the inlined block)
  2. The Superpowers heading (regex `^#{2,4} Superpowers Integration Hooks \(MANDATORY`). When the deprecated H1 is absent, this heading is the start anchor. The ~20 lines of preamble between `## Development Workflow` and the Superpowers heading (the "This project uses three complementary tools" table) are **left in CLAUDE.md** — they're short, often customised by projects, and removing them risks deleting project-owned content.

  Note: `## Development Workflow` is **explicitly NOT** a valid start anchor by itself, because that heading title is generic enough that many projects use it for their own development-workflow content. Anchoring on it would risk eating project-owned headings.

- **End line:** the inlined block from the deprecated template extends through the `## Skill routing` section (lines 161–185 of the source template) — past `13 Red Flags`, past `## GSD Workflow Enforcement`. The end anchor is whichever comes first of:
  1. End of the routing-rules list inside `## Skill routing` (last line containing `→ invoke <name>` followed by a blank line).
  2. The next H1 or H2 heading whose title does NOT match a known template H2 (i.e. NOT one of: `## Development Workflow`, `## Workflow commitment`, `## GSD Workflow Enforcement`, `## Skill routing`, `## Workflow` — the last is added by Step 3 above and must NEVER be eaten by Step 4).
  3. End of file.

  List-item continuation past item 13 of the 13 Red Flags numbered list (e.g. team-added items 14+) is included in the extraction range; the customisation is preserved by user-pick option (b) below.

The runtime then compares the extracted range against
`templates/.claude/claude-md/workflow.md` (canonical baseline). Three
outcomes:

| Outcome | User prompt |
|---|---|
| Extracted range matches canonical (modulo whitespace and heading-level reframing) | "Inlined block detected (pristine copy of the legacy inlined template). Safe to remove — the canonical content is already vendored at `.claude/claude-md/workflow.md`. Proceed?" Default: yes. |
| Extracted range diverges from canonical | "Inlined block detected, but it has been customised relative to the legacy inlined template. Diff: [show]. Choose: (a) Replace with canonical (lose customisations), (b) Vendor the customised block as `.claude/claude-md/workflow.md` (preserve customisations as the vendored copy; the canonical from Step 1 is overwritten), (c) Skip removal (CLAUDE.md keeps the inline copy; vendored file from Step 1 stays as the canonical reference)." |
| Detection ambiguous (markers found but end-anchor cannot be determined with confidence — e.g. inlined block was edited so heavily that no template H2 remains and no clean termination boundary exists) | "Inlined block markers found in CLAUDE.md but the extraction boundary is unclear. Manual review recommended. Skip this step?" Default: yes. |

When the user confirms removal, the runtime deletes the extracted line
range from CLAUDE.md (preserving content above and below the range).

**Rollback:** the migration runtime has not committed yet during 0009's
execution (see Step 2 rollback note). To revert Step 4's CLAUDE.md
mutation, the runtime restores CLAUDE.md from a working-tree snapshot
captured at Step 4 entry (after Step 3's reference-block append, before
Step 4's line-range deletion). If the runtime opts to abort the entire
migration via `update/SKILL.md` Step 5 #8, the rollback chain restores
all of Steps 1–4 to the pre-0009 disk state (the prior migration's
commit). `git checkout HEAD -- CLAUDE.md` would discard ALL of this
migration's edits, including Step 3's reference-block append — use only
when aborting the entire migration, not for per-step rollback.

### Step 5: Bump installed version field in `.claude/skills/agentic-apps-workflow/SKILL.md`

**Idempotency check:** `grep -q '^version: 1.8.0' .claude/skills/agentic-apps-workflow/SKILL.md`
**Pre-condition:** `.claude/skills/agentic-apps-workflow/SKILL.md` exists and currently has `version: 1.6.0`
**Apply:** Edit the file frontmatter to change `version: 1.6.0` → `version: 1.8.0`.
**Rollback:** Edit `version: 1.8.0` → `version: 1.6.0`.

## Post-checks

```bash
# Vendored file exists with sane content (heading-level-agnostic match)
test -f .claude/claude-md/workflow.md
grep -qE "^#{2,4} Superpowers Integration Hooks \(MANDATORY" .claude/claude-md/workflow.md

# CLAUDE.md links to the vendored file
grep -q "claude-md/workflow.md" CLAUDE.md

# Inlined block absence — soft check; Step 4 may have been skipped at user
# request, in which case this check fails and the migration is recorded as
# partial. Not a hard error.
if grep -qE "^#{2,4} Superpowers Integration Hooks \(MANDATORY" CLAUDE.md; then
  echo "WARN: inlined block still present in CLAUDE.md (Step 4 was skipped or partial). Re-run migration 0009 with --migration 0009 to retry the extraction."
fi

# Version bumped
grep -q '^version: 1.8.0' .claude/skills/agentic-apps-workflow/SKILL.md
```

## Skip cases

- **Project not at v1.6.0** — pre-flight blocks. The update skill chains
  earlier migrations first if the project is older.
- **Workflow scaffolder older than 1.8.0** (vendored template source missing)
  — pre-flight blocks with `git pull` instruction.
- **CLAUDE.md missing** — pre-flight blocks (extremely rare; suggests the
  project's workflow is half-installed; user should re-run setup).
- **Step 4 user-skipped** — migration completes with partial outcome;
  vendored file is in place and CLAUDE.md links to it; the inlined block
  remains in CLAUDE.md as duplication. Re-running 0009 with
  `--migration 0009` retries the extraction prompt.

## ADR opportunities

After this migration, the update skill prompts: "Want to draft a project

Default: skip.

## Notes for runtime implementers

The Step 4 detection + extraction is the most novel logic in this
migration. Two safety properties the runtime must enforce:

1. **Never delete content from CLAUDE.md without user confirmation and a
   diff preview.** The per-step Apply prompt (`update/SKILL.md` Step 5
   #5) already gates this; Step 4's three-way pick is presented inside
   that prompt.
2. **The extraction boundary must be precise.** If the runtime cannot
   determine the end line with confidence (e.g. the user has appended
   project-specific content immediately after item 13 of the 13 Red
   Flags list with no separating heading), it must fall back to the
   "Detection ambiguous" outcome and skip rather than guess.

Hand-built fixtures for these scenarios live at
`migrations/test-fixtures/0009/`; see `migrations/run-tests.sh` for the
assertion contract.
