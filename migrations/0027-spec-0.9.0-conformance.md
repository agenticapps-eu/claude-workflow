# Migration 0027 — Spec 0.9.0 conformance claim (v2.4.0 -> 2.5.0)

Raises the installed workflow skill's conformance claim from
`implements_spec: 0.4.0` to `0.9.0`, brings its §04 red-flag block into line
with the composition rules core spec 0.8.0 introduced, and gives it the
**"Spec deltas"** section core spec §09 requires. See ADR-0040.

The claim was incoherent, not the implementation: the skill already shipped §14
wiring (a 0.6.0 section, migration 0023) and §15 wiring (a 0.7.0 section,
migration 0025), while claiming 0.4.0 and declaring no deltas. §09 requires a
host to name every requirement it does not satisfy — with no such section,
neither `full` nor `partial` was honestly claimable.

Four changes reach existing installs:

1. **§04 red flags are reordered.** Core 0.8.0 resolved a contradiction in §04:
   host-specific flags MUST be appended *after* the canonical 13 (position 14+);
   inserting one between canonical flags is now explicitly forbidden. This host
   carried `/gsd-review skipped — no {phase}-REVIEWS.md artifact` at position 8,
   renumbering canonical 8–13 into 9–14. It moves to position 14. The 13
   canonical flags are already byte-identical here, so this is a **reordering,
   not a rewrite** — no flag's wording changes. The heading stays
   `## 14 Red Flags — STOP → DELETE → RESTART` (0.8.0 makes the leading count
   non-normative).
2. The installed workflow skill gains the `## Spec deltas (spec 0.9.0)` section
   and an `implements_spec: 0.9.0` frontmatter line.
3. `.planning/config.json`'s `_enforcement_contract` pointer is repointed from
   `docs/workflow/ENFORCEMENT-PLAN.md` (a path that never existed) to
   `docs/ENFORCEMENT-PLAN.md` — now the single §09 item-3 hook-bindings table —
   and the dangling `observability_scan.programmatic_hook` key is dropped.
4. The dead `observability-postphase-scan.sh` hook is removed (Step 5), but only
   when it is present AND registered in no `settings.json` event.

Fresh installs get all four from the snapshot
(`setup/snapshot/agentic-apps-workflow-SKILL.md` +
`setup/snapshot/planning-config.json`); the drift guard
(`migrations/check-snapshot-parity.sh`) fails if any drops out of the seed.

**Supported upgrade floor:** `2.4.0 -> 2.5.0`. Projects below 2.4.0 replay the
chain through 0026 first.

## Steps

### Step 1 — Reorder the §04 red-flag block (canonical 13 to positions 1–13)

Core 0.8.0 §04: host-specific flags MUST be appended after the canonical 13.
The host flag moves from position 8 to position 14; canonical flags 8–13
reclaim their canonical numbers. **No flag's wording changes** — this rewrites
only the ordering and the leading numbers.

The whole block is replaced with the correct one rather than patched line by
line: a numbered list cannot be reordered safely with per-line seds (every
subsequent number shifts), and a literal block replacement is verifiable by
exact match.

**Idempotency check (positive — already reordered):**
```bash
grep -q '^8\. Two-stage review collapsed into one$' \
  .claude/skills/agentic-apps-workflow/SKILL.md
```

**Pre-condition (the known-bad ordering is present):**
```bash
grep -q '^8\. `/gsd-review` skipped' .claude/skills/agentic-apps-workflow/SKILL.md
```

**Apply:**
```bash
TARGET=.claude/skills/agentic-apps-workflow/SKILL.md

# Replace the 14 numbered lines between the heading and the following blank
# line. Anchored on the canonical heading; the count in it is non-normative
# per 0.8.0 and is left as-is (14 flags total, which remains correct).
awk '
  /^## 14 Red Flags — STOP → DELETE → RESTART$/ { print; inblock=1; next }
  inblock && /^$/ && !emitted {
    print ""
    print "1. Code written before the test (for TDD tasks)"
    print "2. Test added after implementation"
    print "3. Test passes on first run — no RED observed"
    print "4. Cannot explain why the test should have failed"
    print "5. Tests marked for \"later\" addition"
    print "6. \"Just this once\" reasoning"
    print "7. Manual testing claimed as verification evidence"
    print "8. Two-stage review collapsed into one"
    print "9. Framing discipline as \"ritual\" or \"ceremony\""
    print "10. Keeping pre-written code as \"reference\" while writing tests"
    print "11. Sunk-cost reasoning about deleting unverified code"
    print "12. Describing discipline as \"dogmatic\""
    print "13. \"This case is different because...\""
    print "14. `/gsd-review` skipped — no `{phase}-REVIEWS.md` artifact"
    emitted=1; skipping=1; next
  }
  skipping && /^[0-9]+\. / { next }
  skipping && /^$/ { skipping=0; print; next }
  skipping { skipping=0 }
  { print }
' "$TARGET" > "$TARGET.0027.tmp" && mv "$TARGET.0027.tmp" "$TARGET"
```

**Rollback:** restore the pre-0027 ordering (host flag back to position 8):
```bash
# Re-run the same awk with the old ordering, or:
git checkout -- .claude/skills/agentic-apps-workflow/SKILL.md
```

### Step 2 — Insert the Spec deltas section into the installed skill

The section is **extracted from the scaffolder's `skill/SKILL.md`** (single
source of truth) rather than duplicated here, so a migrated install is
byte-identical to a fresh snapshot install and the text cannot drift.

Unlike 0025's ritual tail, this section is **not** last in the file — it sits
immediately before `## Knowledge Capture — Ritual Tail`. So the extraction
bounds on the next heading, and the insert anchors on that heading rather than
appending to EOF.

**Idempotency check (positive — section already present):**
```bash
grep -q '^## Spec deltas (spec 0.9.0)' .claude/skills/agentic-apps-workflow/SKILL.md
```

**Apply:**
```bash
SCAFFOLDER=~/.claude/skills/agenticapps-workflow
TARGET=.claude/skills/agentic-apps-workflow/SKILL.md

# Extract: from the Spec deltas heading up to (not including) the next heading.
awk '/^## Spec deltas \(spec 0\.9\.0\)/{f=1}
     f && /^## Knowledge Capture — Ritual Tail/{exit}
     f' "$SCAFFOLDER/skill/SKILL.md" > "$TARGET.0027.section"

# Insert before the ritual-tail heading (pre-flight 4 guarantees it exists).
awk -v secfile="$TARGET.0027.section" '
  /^## Knowledge Capture — Ritual Tail/ && !done {
    while ((getline line < secfile) > 0) print line
    close(secfile)
    done = 1
  }
  { print }
' "$TARGET" > "$TARGET.0027.tmp" && mv "$TARGET.0027.tmp" "$TARGET"
rm -f "$TARGET.0027.section"
```

### Step 4 — Repoint `_enforcement_contract` and drop the dangling hook ref

### Step 5 — Remove the dead `observability-postphase-scan.sh` hook

The hook ships to `.claude/hooks/` but is registered in **no** `settings.json`
event, so it is unreachable as a lifecycle hook. Migration 0018 in this repo is
a tombstone: the file and its own migration 0018 live in
`agenticapps-observability`, which owns this surface since 2.0.0. Its post-phase
invocation is now the `/observability scan --since-commit` step in CLAUDE.md,
which does not depend on a file this repo ships. `docs/UPGRADING.md` already
states claude-workflow ships no observability scaffolding.

**Removal is conditional and fail-safe.** It removes the file only when it is
BOTH present AND registered in no event of `.claude/settings.json`. A project
that deliberately wired it keeps it — this migration never deletes a live hook.

**Idempotency check (positive — already gone):**
```bash
[ ! -e .claude/hooks/observability-postphase-scan.sh ]
```

**Apply:**
```bash
HOOK=.claude/hooks/observability-postphase-scan.sh
if [ -e "$HOOK" ]; then
  REGISTERED=0
  if [ -f .claude/settings.json ] && jq empty .claude/settings.json 2>/dev/null; then
    if jq -e '[.hooks // {} | .[]? | .[]? | .hooks[]? | .command?]
              | map(select(type == "string"))
              | map(select(test("observability-postphase-scan\\.sh")))
              | length > 0' .claude/settings.json >/dev/null 2>&1; then
      REGISTERED=1
    fi
  fi
  if [ "$REGISTERED" -eq 1 ]; then
    echo "SKIP: $HOOK is registered in .claude/settings.json — leaving it in place."
  else
    rm -f "$HOOK"
    echo "Removed dead hook: $HOOK (registered in no settings.json event)."
  fi
fi
```

**Rollback:** `git checkout -- .claude/hooks/observability-postphase-scan.sh`
(or reinstall via the obs repo's migration 0018, which owns the file).

All five post-checks are hard guarantees.

**This migration changes no enforcement behavior.** It corrects what the
installed skill *claims*, the order in which its red flags are listed, and where
it points; every gate binding, ritual, and *live* hook is untouched. Step 5
removes only a hook that was registered nowhere and therefore never ran.

## Skip cases

- **`from_version` mismatch** (project not at 2.4.0) → migration framework
  skips silently per the standard rule. Projects below 2.4.0 replay 0026 first.
- **Section already present** → Step 2 is a no-op.
- **Red flags already reordered** → Step 1 is a no-op.
- **No `.planning/config.json`, or no `_enforcement_contract` /
  `programmatic_hook` key** → Step 4 is a no-op; the migration still completes
  (the pointer is optional metadata).
- **A project that customized `_enforcement_contract`** to some third path →
  Step 4 leaves it alone; it only rewrites the exact known-bad literal.
- **Hook absent, or hook present but registered** → Step 5 is a no-op.

## Compatibility

- **Additive (minor) bump** to `2.5.0`: no breaking change. Step 1 reorders a
  list without changing any flag's wording; Step 2 only inserts a section;
  Steps 3/6 rewrite single scalar values; Step 4 drops a key that pointed at a
  file registered nowhere; Step 5 removes that file only when it is unreachable.
- **Drift coupling:** as the highest-numbered migration file, 0027's
  `to_version` (2.5.0) becomes the drift target asserted by
  `test_skill_md_version_matches_latest_migration_to_version`; `skill/SKILL.md`
  is bumped to 2.5.0 in lockstep, and `check-snapshot-parity.sh` requires the
  snapshot VERSION to equal it.
- **Rebased onto 0026.** This migration originally targeted `0026` /
  `2.3.0 -> 2.4.0`. `feat/gitnexus-background-reindex` landed first and took
  both that slot and 2.4.0, so this rebased to `0027` / `2.4.0 -> 2.5.0` as the
  plan prescribed. 0026 is immutable and untouched; both migrations' harnesses
  coexist in `run-tests.sh`.
