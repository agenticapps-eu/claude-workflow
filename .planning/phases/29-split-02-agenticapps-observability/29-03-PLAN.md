<xfail_semantics>
The shared harness exits non-zero on ANY FAIL>0. The 4 known-failing 0017 fixtures (02,06,10,11
— FIX-0017 engine scope, deferred) would trip that gate. Encode them as EXPECTED FAILURES:

- Maintain an explicit list in the obs run-tests.sh, e.g.
  `OBS_XFAIL_0017="02 06 10 11"` (the 4 known FIX-0017 fixtures).
- In test_migration_0017's per-fixture loop: when a fixture in the XFAIL list FAILS, increment a
  dedicated `XFAIL` counter and print `XFAIL: 0017/<fixture> (FIX-0017 deferred)` — do NOT
  increment `FAIL`.
- If an XFAIL-listed fixture unexpectedly PASSES, that IS a failure (the known-bug list is
  stale): print `XPASS: 0017/<fixture> (expected-fail but passed — update OBS_XFAIL_0017)` and
  increment `FAIL`.
- A fixture NOT in the XFAIL list that fails increments `FAIL` normally (a real regression).
- Summary prints PASS / FAIL / XFAIL / SKIP. Exit non-zero ONLY if `FAIL>0`. "Green" therefore
  means zero UNEXPECTED failures (XFAIL>0 is allowed and documented).

This keeps the 4 × 0017 known-failures travelling without making the harness lie about "green".
</xfail_semantics>
