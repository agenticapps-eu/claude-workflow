    Title line: `# 0031 — 0019 engine accepts index.ts as canonical anchor`
    Header: `**Status**: Accepted  **Date**: 2026-05-31  **Phase**: 25-fix-0019-engine-and-cron-wrappers`

    Sections:
    - **## Context** — Migration 0019's apply engine (`templates/.claude/scripts/migrate-0019-sentry-crons-and-healthz.sh`) discovered wrapper roots by looking for `lib-observability.ts` (cf-worker, cf-pages) or `observability.go` (go-fly-http) or `index.ts` scoped to `*/_shared/observability/` (supabase-edge). But `add-observability/templates/ts-cloudflare-worker/meta.yaml:target.wrapper_path` is `src/lib/observability/index.ts`, NOT `lib-observability.ts`. The template SOURCE file is named `lib-observability.ts`; the materialised TARGET file is `index.ts`. Same for cf-pages: `functions/_lib/observability/index.ts`. The engine never matched real materialised projects — only the test fixtures, which hand-seeded the SOURCE filename. Issue #56 surfaced this when callbot (v1.16.0 → v1.19.0) attempted to apply 0019 and the engine reported "no wrappers found" with exit 0.
    - **## Decision** — Engine accepts `index.ts` AND `lib-observability.ts` as anchor filenames for `ts-cloudflare-worker` and `ts-cloudflare-pages` stacks. The `middleware.ts` (cf-worker) / `_middleware.ts` (cf-pages) co-anchor requirement remains — guards against unintended matches in unrelated dirs. The engine's content-hash canonicaliser is unchanged: both filenames hash to the same canonical fingerprint because they carry the same content.
    - **## Alternatives Rejected** — 3-row table verbatim from RESEARCH.md (rename template source files / REFUSE on index.ts / hybrid alias-with-warning).
    - **## Consequences** — Engine now recognises any project scaffolded since 1.16.0 (canonical materialised path). Pre-engine-fix workarounds (hand-applied 0019) no longer needed. Test fixtures gain coverage for `index.ts`-anchored wrappers (08, 09). Existing `lib-observability.ts`-seeded fixtures continue to pass — both filenames classify as the same stack. No spec change. Extends ADR-0028.

    **Author `docs/decisions/0032-cron-monitor-generic-narrowing.md`:**

    Title: `# 0032 — withCronMonitor generic narrowed for strict Env`
    Header: same format as ADR-0031.

    Sections:
    - **## Context** — Phase 22's `withCronMonitor<E extends Record<string, unknown>>` requires every consumer Env interface to either add `[key: string]: unknown` (loses index-access safety) or cast at every call site. callbot's strict `Env { SENTRY_DSN: string; SERVICE_NAME: string; SUPABASE_URL: string; ... }` (no index signature) couldn't satisfy the constraint — callbot worked around by `as Record<string, unknown>` at every call site. Issue #56 Finding 3.
    - **## Decision** — Narrow to `<E extends { SENTRY_DSN?: string; SERVICE_NAME?: string }>`. Inside the wrapper, env-key lookup becomes `(env as unknown as Record<string, unknown>)[envKey]` — wrapper internals know env is loose-by-design; callers no longer need to lie. Apply symmetrically to all three TS templates AND the new `queue-monitor.ts` (D-07) AND the bundled openrouter-monitor copy (D-21).
    - **## Alternatives Rejected** — table: (a) keep loose Record<string, unknown> — forces every Env to add index signature; (b) require named-Env-only interface — too prescriptive; (c) add index-sig escape hatch + doc warning — half-fix.
    - **## Consequences** — Strict-typed Env interfaces (callbot, cparx, fxsa) satisfy the constraint without casts. Generic narrowing is functionally non-breaking — every previously-conforming `E` (had index signature, almost certainly had SENTRY_DSN somewhere) still conforms. add-observability bumps 0.8.0 → 0.9.0 (minor, surface expansion). Test surface gains D-16 type-level fixture with strict-typed Env. Synthetic strict-Env fixture (test-fixtures/0019/10-strict-env-typecheck) verifies SC5 acceptance.

    **Author `docs/decisions/0033-with-queue-monitor.md`:**

    Title: `# 0033 — withQueueMonitor + Migration 0021 for Cloudflare Queue consumers`
    Header: same format.

    Sections:
    - **## Context** — Phase 22 shipped `withCronMonitor` for scheduled handlers. Cloudflare Queue consumer handlers (`(batch, env, ctx) => Promise<void>` first-arg `MessageBatch<Body>`) have no parallel wrapper. callbot hand-rolled a `withMonitor` helper for queue handlers in `apps/backend/src/index.ts` — duplicative + drift-prone. Issue #56 Finding 4. Separately, the 0019 migration runner's `from_version: 1.17.0` exact-match contract (migrations/README.md:60-99) means re-revving 0019 cannot deliver `queue-monitor.{ts,go}` to projects already at v1.19.0 (callbot included).
    - **## Decision** — Two complementary deliverables: (i) New `withQueueMonitor` export in three TS templates (cf-worker, cf-pages, supabase-edge) with Guarded Shape A (ADR-0029) semantics. Signature mirrors Cloudflare's canonical `QueueHandler<Env, Message, Props>`. Slug resolution mirrors withCronMonitor D6 3-source policy; auto-derive uses `${SERVICE_NAME ?? "service"}:queue:${batch.queue}`. Default handler name `"queue"`. (ii) New Migration 0021 (`from_version: 1.19.0` → `to_version: 1.20.0`, additive single-file copy) ships `queue-monitor.{ts}` to already-migrated v1.19.0 projects. Idempotency marker: `queue-monitor.ts` presence. Per pitfall 2 option (d).
    - **## Alternatives Rejected** — table: (a) re-shape withCronMonitor to accept MessageBatch OR ScheduledController — Phase 22 D1 separate-wrapper precedent applies; (b) extend cron's name — confusing; (c) ship queue-monitor in 0019 only — runner contract prevents reaching v1.19.0 projects; (d) skip migration, document manual copy — friction; (e) add --force flag — doesn't exist today, expands engine surface for one-off path.
    - **## Consequences** — Cloudflare Queue consumer handlers now have heartbeat parity with scheduled handlers. callbot can replace its local helper with upstream `withQueueMonitor`. v1.19.0 projects pick up the wrapper via 0021 (no manual recovery required). claude-workflow bumps 1.19.0 → 1.20.0 (minor, two migration deltas: 0019 re-rev + new 0021). add-observability bumps 0.8.0 → 0.9.0 (minor, three template surface changes: D-03/D-05/D-07). Multi-queue handlers dispatching on `batch.queue` MUST set `monitorSlug` explicitly (silent + docs policy, mirrors Phase 22 D11). Extends ADR-0029.
