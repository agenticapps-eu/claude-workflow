# Spec 0.7.0 conformance — design

**Date:** 2026-07-14
**Status:** Approved
**Scope:** claude-workflow (4 changes) + agenticapps-workflow-core (3 changes)

## Context

An audit of claude-workflow against `agenticapps-workflow-core` spec 0.7.0 found
the implementation substantively complete but the conformance *claim* incoherent,
plus three real defects. Three sources disagree on the spec version:

| Source | Claims |
|---|---|
| Core spec | 0.7.0 |
| `reference-implementations/README.md` (core ledger) | claude-workflow @ 0.3.0, `full` |
| `skill/SKILL.md:4` | `implements_spec: 0.4.0` |
| Actually wired | 0.7.0 (§15 ritual tail, §14 delegated, plan-review gate) |

The 0.4.0 stamp is deliberate, not drift — ADR-0038 records that it "tracks the
last full-conformance audit," and that bumping to 0.7.0 required auditing the
sections added at 0.5.0/0.6.0. That audit is now done; this is the bump.

## Defects to close

### D1 — §11 never reaches fresh installs (the only user-visible defect)

The canonical §11 "Coding Discipline" block lives at
`templates/spec-mirrors/11-coding-discipline-0.4.0.md`, byte-identical to the
spec's canonical block (verified: 79 lines, zero diff; the mirror was kept in
sync — core `10f2c96` → host `34ee72e`).

It is reachable **only** by replaying migration 0014. Since ADR-0036,
`/setup-agenticapps-workflow` installs the snapshot and replays nothing.
`setup/`, `bin/build-snapshot.sh`, and `migrations/check-snapshot-parity.sh`
contain zero references to `spec-mirrors`. So greenfield projects never receive
§11 — while `setup/snapshot/MANIFEST.md:4` asserts the snapshot is "the same
files a full `0000`→latest migration replay would produce."

Reproducing §11 verbatim in the primary project-instruction file is a §09 item-1
MUST for `full`. The parity guard checks §15's trigger wiring but not §11, so CI
cannot catch this.

### D2 — `design-critique` fires on the inverted condition

Spec §02 triggers `design-critique` on a UI plan **with an existing**
UI-SPEC.md. `templates/config-hooks.json:28` has
`"trigger": "ui_hint_yes && design_shotgun_completed"`, but `design_shotgun`'s
own trigger is `"ui_hint_yes && no_ui_spec_yet"`. When a UI-SPEC.md already
exists, shotgun does not fire, so critique never fires — exactly the spec's
trigger condition.

### D3 — `check-hooks.sh` verifies 4 of 9 hooks

`bin/check-hooks.sh:24` loops `database-sentinel design-shotgun-gate
skill-router-log session-bootstrap`. Nine hooks are registered in
`templates/claude-settings.json`. `multi-ai-review-gate.sh` — the plan-review
enforcer — is unverified, the same silent-non-firing class of bug ADR-0025 closed.

## Design

### C1 — §11 via the snapshot path

`bin/build-snapshot.sh` copies `templates/spec-mirrors/` →
`setup/snapshot/spec-mirrors/`. A new setup step reads
`$SNAP/spec-mirrors/11-coding-discipline-0.4.0.md` and injects it into CLAUDE.md
behind the same provenance anchor migration 0014 uses:

```
<!-- spec-source: agenticapps-workflow-core@0.4.0 §11 -->
```

at the same insertion point — immediately before the first `## ` heading. Output
is byte-identical to the replay path, so the MANIFEST contract holds.

**Setup needs two of 0014's three branches, not all three.** Setup is fail-closed
on re-runs (it refuses on an existing `.claude/skills/agentic-apps-workflow/` and
routes to `/update`), so a CLAUDE.md carrying *our* provenance anchor is
unreachable on the setup path. The stale-provenance `replace` branch — ~30 lines
of awk — is dead code here. Setup implements:

- §11 heading present **without** provenance → **abort** (hand-pasted block;
  the same refusal 0014 makes)
- otherwise → **insert** before the first `## ` heading

Migration 0014 is left **immutable**. Shipped migrations are not retroactively
edited, so no shared-injector refactor: ~10 lines of duplication is the correct
trade against mutating history.

`spec-mirrors/` becomes a new snapshot directory. That widens the snapshot's
surface slightly — the accepted cost of keeping replay and snapshot in agreement.

**Alternatives rejected:**

- **Inline §11 into `claude-md-workflow.md`.** Cheapest, but §11 requires the
  block in the *primary* project-instruction file (CLAUDE.md), not a vendored
  sub-file; and it diverges from 0014's output, breaking the MANIFEST contract
  and adding a third divergent prose copy — already a finding in this audit.
- **Setup shells out to migration 0014.** Violates ADR-0036 ("no migration
  executes" in setup) and depends on the `$HOME` scaffolder clone.

### C2 — `design-critique` trigger

`"trigger": "ui_hint_yes && ui_spec_exists"`, matching §02.

### C3 — `check-hooks.sh` covers all 9 hooks

Derive the hook list from `templates/claude-settings.json` rather than hardcoding,
so a newly registered hook cannot silently escape verification.

### C4 — Single hook-bindings table, 0.7.0 claim, Spec deltas

`docs/ENFORCEMENT-PLAN.md` is promoted to the single §09 item-3 table: add the
missing `impeccable-audit` and `db-pre-launch-audit` rows, de-nest
`database-security` from `security`'s `sub_gates` (§02 forbids merging two gates
into one), restore canonical gate names, and add the §13 `ts-declare-first`
binding (§13 Conformance SHOULD).

`skill/SKILL.md` gets `implements_spec: 0.7.0` and a **Spec deltas** section —
the repo currently has none anywhere, which is what makes the claim incoherent
rather than merely imprecise.

Deltas to document:

| Delta | Rationale | `full` preserved? |
|---|---|---|
| §14 prompt-injection | Scaffolder builds no LLM prompts from non-self-authored values → trivially conformant per §09; generator delegated to `injection-guard` (obs 0.13.0) | Yes — §09 requires only that it be *said* |
| §13 implicit GSD trigger | Skill ships; explicit trigger works; implicit detection deferred and tracked. §13's Conformance section is SHOULD/MAY throughout | Yes |
| §10 observability | Delegated to `agenticapps-observability` 0.13.0; fails closed via migration 0022 pre-flight | Yes — a satisfied MUST per §09, not a delta, but recorded for clarity |

Also fix the dangling `docs/workflow/ENFORCEMENT-PLAN.md` pointer
(`skill/SKILL.md:25,206`) — the file is at `docs/ENFORCEMENT-PLAN.md`.

### C5–C7 — Upstream fixes in agenticapps-workflow-core

- **C5 — §09 gate count.** `spec/09-conformance.md:61` says "Section 02
  enumerates 15 gates." It enumerates **16** (verified by `grep -c '^#### '`);
  the count was never updated when `plan-review` landed at 0.5.0.
- **C6 — §08 amendment.** `spec/08-migration-format.md:193` requires migrations
  live in "a single directory consumed by **both** setup and update flows."
  ADR-0036 deliberately reversed this for the setup path with a real mitigation
  (`check-snapshot-parity.sh` in CI). The snapshot approach won on merit and
  codex-workflow ships the same pattern, so **amend §08** to recognize
  snapshot-install as a conformant setup strategy where a parity guard proves
  equivalence — rather than record a permanent delta against a MUST the
  reference implementation knowingly rejects.
- **C7 — Ledger row.** Refresh claude-workflow's
  `reference-implementations/README.md` row to 0.7.0 / `full`. The current row
  describes a world that no longer exists: it credits `add-observability` as
  shipping from claude-workflow, deleted in the 2.0.0 split (`217baec`).

## Out of scope

- Wiring §13's implicit GSD trigger (documented as a delta; own phase).
- The 7 single-stage REVIEW.md artifacts in `.planning/phases/` (historical
  artifacts of this repo's own process, not the shipped product).
- The §08 structural migration linter (real gap; own phase — would mechanically
  catch the missing pre-conditions in 0005/0016/0024/0025/0026).
`update/SKILL.md:20-23` still asserts the ADR-0036-superseded "setup replays
migrations" stance. It is a four-line prose correction on a claim this design
makes demonstrably false, so it lands in C4 rather than being deferred.

## Versioning

Engine/doc fixes to conformance bookkeeping do not add a migration, so no
scaffolder version bump is implied by C2/C3/C4 alone. C1 changes what a fresh
install lays down — it ships a new snapshot artifact and a new setup step — so
it takes a minor bump and a CHANGELOG entry: **2.3.0 → 2.4.0**. Existing
installs already have §11 via 0014; no new migration is needed to backfill them.

## Branch and merge order

This work branches off `main` (2.3.0), **not** off the in-flight
`feat/gitnexus-background-reindex` (2.4.0, awaiting `/cso` + PR). D1 is a live
defect in every fresh install; stacking would block it behind 0026's remaining
gates.

The two branches overlap on `bin/build-snapshot.sh`,
`migrations/check-snapshot-parity.sh`, `skill/SKILL.md`, `CHANGELOG.md`, and
`setup/snapshot/`. Every overlap is additive — each branch appends a `cp` line
to the builder and a section to the parity guard — so the conflicts are
mechanical. Both branches claim 2.4.0: **whichever merges second rebases to
2.5.0** and renumbers its CHANGELOG entry. Noted here so the second PR's author
does not mistake the collision for a mistake.
