# ADR-0040: Claim spec 0.8.0 honestly — §04 reordering, one bindings table, five fixes, four deltas

**Status**: Accepted  **Date**: 2026-07-14  **Linear**: —
**Core contract**: `agenticapps-workflow-core/spec/` @ 0.8.0 (§02, §04, §08,
§09, §10, §11, §13, §14, §15)

## Context

A conformance audit of this repo against the core spec found the conformance
**claim** incoherent — not the implementation. Four separate artifacts
disagreed about what this host implements:

- `skill/SKILL.md` frontmatter said `implements_spec: 0.4.0`, while the repo
  actually ships §14 wiring (a 0.6.0 section, via migration 0023) and §15
  wiring (a 0.7.0 section, via migration 0025 / ADR-0038).
- Core's `reference-implementations/README.md` ledger row said `0.3.0`.
- **No "Spec deltas" section existed anywhere.** §09 requires one for any
  unsatisfied requirement, at both `full` and `partial`. Without it, neither
  level was honestly claimable: `full` asserts there is nothing to declare,
  and this host has things to declare.

The audit also found three real defects behind the claim, plus a structural
one:

1. **§11's canonical block never reached the setup path.** §09 item 1 requires
   canonical-prose blocks to be reproduced verbatim in the host's instruction
   file. The block existed at `templates/spec-mirrors/11-coding-discipline-0.4.0.md`
   but only the update path injected it — fresh installs silently omitted it.
2. **The `design-critique` gate had an inverted trigger.** §02 fires it on a UI
   plan *with* an existing UI-SPEC.md; the host fired it after `design-shotgun`,
   i.e. when no UI-SPEC.md existed yet.
3. **`bin/check-hooks.sh` verified a hardcoded four hooks.** The plan-review
   enforcer (`multi-ai-review-gate.sh` — the §02 `plan-review` gate, ADR-0025)
   was not among them and was therefore silently unchecked.
4. **Hook bindings were scattered and contradictory.** §09 item 3 requires the
   binding for every applicable gate in **a single** host-side table. Bindings
   were spread across `docs/ENFORCEMENT-PLAN.md` (missing two gates, merging a
   third), `templates/config-hooks.json` (the only file listing all gates, but
   whose own `_note` says orchestrator code does not read it), and
   `skill/SKILL.md`. `skill/SKILL.md` pointed at `docs/workflow/ENFORCEMENT-PLAN.md`
   — a path that does not exist.
5. **The §04 red-flag block violated the 0.8.0 composition rule.** Core 0.8.0
   (released 2026-07-14, mid-flight for this work) resolved a contradiction in
   §04: the section had required verbatim reproduction of the canonical block
   *and* permitted host additions — two rules no host could satisfy at once.
   0.8.0 scopes what "verbatim" binds: host-specific flags MUST be appended
   after the canonical 13 (position 14+), and the canonical 13 keep positions
   1–13 with the listed wording. Core's changelog names this host's violation
   by name: `skill/SKILL.md` carried `/gsd-review skipped — no
   {phase}-REVIEWS.md artifact` at position **8**, renumbering canonical 8–13
   into 9–14.

A fifth defect surfaced while auditing the hook set:

6. **`observability-postphase-scan.sh` was a dead hook.** It shipped to
   `.claude/hooks/` on both install paths but was registered in **no**
   `settings.json` event — only as `programmatic_hook` metadata inside
   `.planning/config.json`, which that file's own `_note` says orchestrator
   code does not read. Migration 0018 here is a tombstone; the file and its
   real migration 0018 live in `agenticapps-observability`, which has owned
   this surface since 2.0.0. Defect 3's new dead-hook check therefore failed
   (exit 1) on every fresh install.

## Decision

**Claim `implements_spec: 0.8.0` at version 2.5.0, and make the claim true**
rather than lowering it to match the weakest artifact. The wiring for §14 and
§15 already shipped; the 0.4.0 claim understated the repo. Four host changes
close the gap:

1. **§11 on the setup path** (Task 1). `setup/SKILL.md` injects the §11 mirror
   from the snapshot; `check-snapshot-parity.sh` asserts the mirror stays
   byte-identical to its `templates/` source and carries the canonical heading.
   The block is byte-identical to the spec's (79 lines), verified.
2. **`design-critique` trigger corrected** (Task 2) to `ui_hint_yes &&
   ui_spec_exists`, matching §02. Guarded by a parity assertion.
3. **`check-hooks.sh` derives its hook set from `settings.json`** (Task 3)
   rather than a hardcoded list, and additionally verifies each hook's event
   binding and flags on-disk-but-unregistered dead hooks. A newly registered
   hook can no longer escape verification.
4. **`docs/ENFORCEMENT-PLAN.md` is promoted to THE §09 item-3 table** (this
   task): all 16 canonical §02 gates, each with trigger, bound skill, and
   required evidence. Canonical gate names live in a `Gate` column and host key
   names in a separate `Host key` column, so the canonical name is never
   ambiguous. `database-security` is de-nested into its own row.
5. **The §04 block is reordered** (this task): the host flag moves to position
   14 and the canonical 13 reclaim positions 1–13. Core calls this "a
   reordering, not a rewrite" — the 13 canonical flags were already
   byte-identical here, so **no flag's wording changes**. The heading stays
   `## 14 Red Flags — STOP → DELETE → RESTART`; 0.8.0 makes the leading count
   non-normative and the rest of the heading is already canonical. Byte-identity
   of the canonical 13 against `spec/04-red-flags.md` is verified.
6. **The dead hook is removed from this repo's payload** (this task):
   `templates/.claude/hooks/observability-postphase-scan.sh` is deleted, the
   dangling `programmatic_hook` reference is dropped from
   `templates/config-hooks.json`, and the CLAUDE.md post-phase step now invokes
   `/observability scan --since-commit` — the obs skill directly — rather than a
   file this repo no longer ships. Migration 0027 removes it from existing
   projects, but only when it is present AND registered nowhere.

**Add a "Spec deltas" section to `skill/SKILL.md`** naming four items: §13's
unwired implicit trigger, §14's trivial conformance, §10's delegation, and the
§08 divergence. §13 and §14 preserve `full`.

### Why `full` survives §13 and §14

- **§13** is SHOULD/MAY throughout its own Conformance section, and explicitly
  permits a host whose primary target is not TypeScript to decline the skill
  entirely with a recorded delta. This host ships the skill anyway; only the
  implicit GSD trigger is missing. It has no `package.json` or `tsconfig.json`,
  so it is not itself a TS project.
- **§14** applies only to hosts whose projects build LLM prompts from
  non-self-authored values. This scaffolder builds none, so the trigger cannot
  occur — §09 calls this trivially conformant and requires only that the host
  say so.

Neither is a MUST this host fails. A delta is the disclosure §09 asks for, not
an admission of non-conformance.

### §08 is a real, open delta — not a resolved one

§08's Conformance requires migrations be "stored in a single directory consumed
by both setup and update flows". ADR-0036 replaced replay-on-setup with a
prebuilt snapshot; setup consumes the snapshot, not the chain.

The intended resolution is to amend §08 upstream to recognise
snapshot-plus-parity-guard as an equivalent consumption mechanism. **That
amendment has not happened.** As of core 0.8.0 (released 2026-07-14, which
changed §04 only), `spec/08-migration-format.md` is unchanged since 0.1.0 and
says nothing about snapshots or parity guards. The delta is therefore recorded
as open and unresolved. Recording it as "amended upstream in 0.8.0" would have
been a false claim in the very section whose purpose is honest disclosure —
and a conformance section that lies is worse than none.

## Alternatives Rejected

**Lower the claim to 0.4.0 and delete the §14/§15 wiring talk.** Rejected: the
wiring ships and works. The claim was wrong, not the code. Lowering it would
make the ledger accurate and the repo undiscoverable for two sections it
actually implements.

**Claim `partial` instead of `full`.** Rejected: `partial` is for unsatisfied
declarative MUSTs and unbound applicable gates. After Tasks 1–3, this host has
neither. §13/§14 are SHOULD-level and trigger-inapplicable respectively; §08 is
a single documented divergence that `full` accommodates via a delta. `partial`
would understate the host as badly as `0.4.0` did.

**§11 — inline the canonical block into `templates/claude-md-workflow.md`.**
Rejected: two copies of a byte-identical canonical block drift. The mirror file
stays the single source and both paths inject from it.

**§11 — have setup shell out to migration 0014.** Rejected: it reintroduces
exactly the replay coupling ADR-0036 removed, for one step, and makes setup
depend on a historical migration remaining replayable forever.

**Keep `templates/config-hooks.json` as the bindings table.** Rejected: it is
the most complete list, but its own `_note` states no orchestrator code reads
it, and JSON cannot carry the trigger/evidence prose §02 expects. It is now
documented as the *seed* the table indexes into, not a second table.

**De-nest `database-security` in `config-hooks.json` too.** Rejected for this
change: §02's prohibition on merging gates binds the *documentation table*, and
the nesting is inert host data. Restructuring the config schema would alter the
snapshot's `planning-config.json` shape and touch migration 0025's fixtures —
a behavior change that belongs in its own migration, not in a docs task. The
table states explicitly that the nesting carries no conformance meaning.

## Consequences

- `skill/SKILL.md` is at 2.5.0 / `implements_spec: 0.8.0`. Core's ledger row
  (a sibling task) cites this claim; the row's `0.3.0` becomes stale until it
  lands.
- `docs/ENFORCEMENT-PLAN.md` is now load-bearing for conformance. Adding a gate
  binding anywhere else without adding it there re-creates the scattered state
  this ADR closes.
- The §08 delta stays open until core amends §08. If the amendment is rejected,
  this host carries the delta permanently — which `full` permits, provided the
  delta is disclosed.
- §09 item 3 says §02 "enumerates 15 gates". §02 actually defines **16** (a
  sibling task fixes the miscount upstream). The table binds all 16; a reader
  comparing counts against §09's prose will see an off-by-one that is §09's,
  not this host's.
- One known gap is documented but deliberately not fixed here, as it needs its
  own change: the §13 implicit trigger.
- **Migration 0027, not 0026.** This work originally targeted migration `0026` /
  `2.3.0 -> 2.4.0`. The sibling `feat/gitnexus-background-reindex` branch
  (ADR-0039) landed first and took both that slot and 2.4.0, so this rebased to
  `0027` / `2.4.0 -> 2.5.0` — the remedy the plan prescribed for whichever
  branch merged second. 0026 is immutable and untouched; both harnesses coexist
  in `run-tests.sh`.
- **`build-snapshot.sh` now prunes `hooks/`, `scripts/`, and `spec-mirrors/`
  before repopulating them.** Those dirs are assembled purely by `cp` globs, so
  the old build only ever ADDED: deleting the dead hook from `templates/` left
  it in the committed snapshot, `--check` reported DRIFT, and a plain rebuild
  could never converge. Found by this change; the prune makes deletions
  propagate.
- **A divergent copy of the §04 block survives outside the instruction file.**
  `templates/claude-md-sections.md` and `templates/.claude/claude-md/workflow.md`
  carry their own 13-flag list with a reworded heading (`13 Red Flags — Trigger
  Automatic STOP → DELETE → RESTART`) and reworded flags (e.g. "Any 'just this
  once' reasoning"). §09 item 1 binds the canonical block to *the host's
  primary instruction file* — `skill/SKILL.md`, the file carrying
  `implements_spec` — and that file is now conformant, so the 0.8.0 claim
  stands. But those copies are what agents actually read at runtime in a
  consuming project. Reconciling them touches the CLAUDE.md payload and needs
  its own migration; flagged, not fixed.
